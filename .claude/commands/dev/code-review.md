# 代码审查

对当前暂存区（git staged）的代码变更进行全面审查。

## 审查步骤
1. 获取暂存区变更：`git diff --cached`
2. 按以下维度逐一审查
3. 输出结构化审查报告

## 审查维度

### 1. 代码质量
- 命名规范：类名大驼峰、方法/变量小驼峰、常量全大写
- 代码可读性：逻辑是否清晰、嵌套是否过深（>3层需重构）
- 重复代码：是否可提取公共方法
- 方法长度：单个方法是否超过50行
- 类职责：是否符合单一职责原则

### 2. 项目规范（hxq-eap特定）
- Service实现类：是否继承正确的基类，事务注解是否正确（get*/select*只读，其他可写）
- 状态码：是否使用标准状态码（0=默认, 1=启用, 2=禁用, 99=删除）
- 错误码：是否使用4位格式（2000=成功, 4***=参数错误, 5***=程序错误）
- 数据库：mapper是否放在正确的包（mapper.eap或mapper.hxq）
- DTO位置：是否按子域组织（domain.dto.eap等）
- 日志：是否使用Log4j2而非默认日志

### 3. 安全性
- 输入验证：Controller参数是否有@Valid/@Validated注解
- SQL注入：MyBatis是否使用#{}而非${}（除非必要）
- XSS防护：返回HTML内容是否转义
- 敏感数据：密码/token是否加密存储，日志是否脱敏
- 权限校验：是否有适当的权限注解或拦截器

### 4. 性能
- N+1查询：是否在循环中执行数据库查询
- 批量操作：是否使用批量插入/更新而非循环单条
- 缓存使用：高频查询是否考虑Redis缓存
- 集合操作：是否使用合适的数据结构（ArrayList vs LinkedList）
- 事务范围：事务是否包含耗时操作（如HTTP调用、文件IO）

### 5. 异常处理
- 异常捕获：是否捕获了具体异常而非Exception
- 异常传播：是否正确抛出业务异常
- 资源释放：IO/连接是否在finally或try-with-resources中关闭
- 日志记录：异常是否记录完整堆栈

### 6. 测试
- 单元测试：新增public方法是否有对应测试
- 边界测试：是否覆盖null、空集合、边界值
- Mock使用：是否正确mock外部依赖

### 7. 文档与注释
- 类注释：是否有@author和功能说明
- 方法注释：public方法是否有JavaDoc（参数、返回值、异常）
- 复杂逻辑：是否有必要的行内注释说明why而非what
- Swagger注解：Controller是否有@Api/@ApiOperation注解

## 输出格式

### 审查摘要
- 变更文件数：X个
- 新增代码行：+X
- 删除代码行：-X
- 主要变更类型：[新功能/Bug修复/重构/配置变更]

### 必须修改 (Blocking) 🚫
> 影响功能正确性、安全性或违反强制规范的问题

- **[问题类别]** 问题描述
  - 位置：`文件路径:行号`
  - 当前代码：`代码片段`
  - 问题原因：具体说明
  - 修改建议：提供可执行的代码示例

### 建议修改 (Suggestion) 💡
> 可提升代码质量但不影响功能的改进点

- **[问题类别]** 问题描述
  - 位置：`文件路径:行号`
  - 改进方向：具体建议
  - 优先级：[高/中/低]

### 表扬 (Praise) ✅
> 值得肯定的优秀实践

- 做得好的地方及原因

### 审查结论
- [ ] 通过，可以合并
- [ ] 需修复Blocking问题后重新审查
- [ ] 建议修复Suggestion后合并
